# Security Policy

## Overview

This document describes the security policy for this project. The goal of this policy is to provide a general guideline for how security issues should be handled when they are discovered in the project. However, this policy is intentionally minimal and does not guarantee that vulnerabilities will be addressed, fixed, or even acknowledged.

Users of this software should understand that security is not a primary focus of this project, and therefore the project maintainers cannot guarantee the confidentiality, integrity, or availability of any systems using this software. By using this project, you acknowledge that you assume all risks related to potential security vulnerabilities.

## Supported Versions

Currently, no versions of this project are officially supported from a security perspective. Security patches may or may not be provided, and there is no commitment to maintaining or updating any released version.

| Version         | Supported |
| --------------- | --------- |
| All versions    | ❌         |
| Future versions | ❓         |

Because of this, users should assume that all versions may contain unpatched vulnerabilities.

## Reporting a Vulnerability

If you discover a security vulnerability, you may report it, but there is no guarantee that the report will be reviewed or responded to. Reports can be submitted through any of the following methods:

* Opening a public GitHub issue
* Mentioning the vulnerability in a pull request
* Posting about it in discussions
* Contacting maintainers if contact information is available

Private disclosure channels are not guaranteed. Vulnerabilities reported publicly may remain visible to everyone and may not be removed.

## Response Time

There is no guaranteed response time for security reports. Maintainers may respond quickly, slowly, or not at all depending on availability, interest, and project priorities.

Possible outcomes include:

* The issue being ignored
* The issue being closed without explanation
* The issue being acknowledged but not fixed
* The issue being fixed eventually
* The issue being fixed accidentally during unrelated changes

## Security Updates

Security updates are not scheduled and may never occur. If updates do happen, they will appear in normal commits without any special notification. There will not necessarily be:

* Security advisories
* CVE registrations
* Coordinated disclosure timelines
* Backported patches

Users are responsible for monitoring the repository and determining whether updates are relevant to them.

## Best Practices for Users

Because the project does not actively maintain strict security practices, users are encouraged to:

* Review all source code before using it
* Run the software only in isolated or experimental environments
* Avoid using the project in production systems
* Implement their own security protections if necessary

Users should assume that vulnerabilities may exist and may remain unfixed indefinitely.

## Liability

The maintainers of this project are not responsible for any damages, data loss, security breaches, or other consequences resulting from the use of this software. This includes, but is not limited to:

* System compromise
* Data corruption
* Unauthorized access
* Service outages

Use this software at your own risk.

## Changes to This Policy

This security policy may change at any time without notice. Future versions of this policy may become more strict, less strict, or be removed entirely.
